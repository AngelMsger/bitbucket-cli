# Safety modes: `--dry-run` and read-only

`bitbucket-cli` gives you two independent guardrails before any mutation:

- **`--dry-run`** — preview the HTTP request the command *would* send.
- **Read-only mode** — block every write for the session.

They compose: `BITBUCKET_CLI_READ_ONLY=1 bitbucket-cli pr merge <ref>
--dry-run` is a useful, byte-identical preview that cannot accidentally
merge anything.

## `--dry-run`

Every mutating command accepts `--dry-run`. The flag short-circuits the
HTTP call and prints the planned request:

```bash
$ bitbucket-cli pr merge myws/myrepo/42 --dry-run
{
  "method": "POST",
  "url": "https://api.bitbucket.org/2.0/repositories/myws/myrepo/pullrequests/42/merge",
  "payload": { "merge_strategy": "merge_commit" }
}
```

Commands that accept `--dry-run`:

| Subtree | Commands |
|---|---|
| `pr` | `create`, `update`, `approve`, `unapprove`, `request-changes`, `decline`, `merge` |
| `comment` | `add`, `update`, `delete` |
| `branch` | `create`, `delete` |
| `repo` | `create`, `delete` |

The preview is generated by the same build helper the live write uses, so
the URL and payload cannot drift from what would actually be sent. Data
Center routes that require a version number (e.g. comment update/delete,
PR decline/merge) still perform the safe read-only GET that fetches that
version — the preview reflects the resulting `?version=N`.

`--dry-run` is **not blocked by read-only mode**. It never sends HTTP, so
it is the recommended way to inspect a write without flipping any switch.

## Read-only mode

Read-only mode is a session-level switch. When active, every mutating
client method (and `pr fetch/checkout --exec`, which would shell out to
`git`) is blocked **before** the HTTP request is built.

### Enabling

Either set it in config:

```yaml
# ~/.angelmsger/bitbucket/config.yaml  (legacy: ~/.bitbucket/config.yaml)
defaults:
  read_only: true
```

Or via environment for an ad-hoc session:

```bash
export BITBUCKET_CLI_READ_ONLY=1
```

Environment beats config when both are set.

### Blocked operation example

```bash
$ bitbucket-cli pr approve myws/myrepo/42
{
  "error": {
    "category": "permission",
    "code": "READONLY_BLOCKED",
    "message": "operation \"ApprovePR\" blocked: read-only mode is enabled",
    "hint": "Re-run with --allow-writes to permit writes for this invocation, or unset BITBUCKET_CLI_READ_ONLY / defaults.read_only.",
    "next_steps": [
      "Add --allow-writes to the command line",
      "unset BITBUCKET_CLI_READ_ONLY",
      "Set defaults.read_only=false in ~/.angelmsger/bitbucket/config.yaml"
    ],
    "retryable": false
  }
}
# exit code: 5  (category=permission)
```

### Per-invocation override: `--allow-writes`

The root-level `--allow-writes` persistent flag overrides the posture for
exactly one invocation:

```bash
BITBUCKET_CLI_READ_ONLY=1 bitbucket-cli --allow-writes pr approve myws/myrepo/42
```

This is the only knob; there is no `--read-only` flag. The asymmetry is
deliberate: persistent posture flows from config/env, per-call overrides
flow from a single flag, and the two cannot conflict.

### Scope

| Layer | Blocked? |
|---|---|
| Bitbucket API writes (POST/PUT/DELETE/PATCH) | Yes |
| `pr fetch/checkout --exec` (mutates local git refs/worktree) | Yes |
| `pr fetch/checkout` without `--exec` (print-only) | No |
| `file get --output <path>` (writes remote content to a local file) | No |
| `config init`, `auth login|logout`, `config use-context` | No |
| `skill install / uninstall` | No |

The boundary is "remote Bitbucket service + your working tree".
`bitbucket-cli`'s own configuration is intentionally outside the scope —
otherwise an agent that turned on read-only would lose the ability to
turn it back off.

## Cookbook

```bash
# Safe review session — read everything, mutate nothing.
export BITBUCKET_CLI_READ_ONLY=1
bitbucket-cli pr status myws/myrepo/42
bitbucket-cli pr files myws/myrepo/42
bitbucket-cli pr threads myws/myrepo/42
# A misclick on a write call is now a clean 5/READONLY_BLOCKED:
bitbucket-cli pr merge myws/myrepo/42 --yes   # blocked, never sent

# Inspect the planned merge without leaving read-only:
bitbucket-cli pr merge myws/myrepo/42 --dry-run

# When you do want to send it, one-shot override:
bitbucket-cli --allow-writes pr merge myws/myrepo/42 --yes
```
